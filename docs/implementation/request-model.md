# 请求模型与流程策略

本文说明 ASC 请求、分页、报表下载和媒体上传的设计策略。Apple 行为核实于 **2026-06**，来源见文末；具体接口与字段以生成的 API 契约为准。

## 请求设计目标

- 业务模块只表达"要做什么"，不拼接底层请求细节。
- 请求层统一处理认证注入、查询表达、错误归一和限流信息。
- 契约类型在每个调用点约束参数与响应，类型不匹配在开发期暴露。
- 需要状态推进或文件处理的流程，不放进普通资源请求路径。

## 普通请求模型

请求内核基于 openapi-fetch 消费生成的契约类型（设计判断见[架构总览](../architecture/overview.md)），认证经 middleware 接缝注入（见[认证实现策略](authentication.md)）。统一模型：

1. 业务模块给出资源、动作和业务参数。
2. 请求层依契约类型组织请求并注入凭证。
3. ASC 返回资源结果或错误。
4. 请求层把结果整理为业务模块可消费的形式，错误归一为语义分类。

ASC 的查询表达是 JSON:API 风格（`filter`、`fields`、`include`、`sort`、`limit` 等），由请求层统一规范，不由各业务模块自行拼接。

## 错误归一

ASC 错误响应是 JSON:API 风格的 `errors` 数组，每项必含 `code`、`status`、`title`、`detail`，可选指向请求体路径或查询参数的 `source`。请求层保留这些原始语义，并归一为分类错误：

| 错误类别 | 识别与处理策略 |
|---|---|
| 认证失败 | 提示凭据或凭证状态问题，必要时触发一次受控重签 |
| 权限不足 | 提示需要更高 ASC 角色或团队 key，不伪装成资源不存在 |
| 限流 | 429（`RATE_LIMIT_EXCEEDED`），保留限流信息，由上层决定退避或终止 |
| 参数不合法 | 借助 `source` 指向调用输入或契约不匹配处 |
| 外部服务失败 | 保留 ASC 响应上下文 |
| 文件处理失败 | 区分下载、解压、解析、上传或提交阶段 |

## 限流与重试

ASC 按 key 限流（默认每小时 3500 次，滚动窗口），每个响应携带 `X-Rate-Limit` 头暴露配额与余量；429 不带 `Retry-After`。策略：

- 请求层在每个响应上解析并透传限流余量，供上层在长任务（批量读取、轮询）中主动节流。
- 重试采用手写的小型退避：仅对 429、5xx 和网络类错误重试，指数退避加抖动，次数有上限；这是可单测的纯逻辑，不引入重试库。
- 重试是请求层的默认行为，但长时间等待属于任务级决策——工作流层可以选择把限流上抛给用户而不是静默等待。

## 分页策略

ASC 用游标分页：响应的 `links.next` 携带不透明 cursor，单页 `limit` 上限一般为 200。分页是控制流，不是单个请求的属性：

1. 读取第一页，产出当前页数据。
2. 存在 `links.next` 则原样跟随，不解析、不构造 cursor。
3. 没有下一页标记时结束；任一页面失败时保留失败页的上下文——已读页数与条目数作为切面附着在原错误上一并上抛，错误分类与原始 errors 不受影响。

业务模块不接触游标，只声明需要单页、限量还是全量数据。多页读取以限流余量地板保护节奏：分页层逐页解析限流余量，仍有后续页且余量低于可配置地板值时主动停止，按限流语义上抛并携带读取进度；已花配额换来的页面在停止前照常交付。默认地板值在小时配额中留出交互与重试余量，可按调用调整或置零关闭。不做静默自适应延时——与「限流与重试」一节口径一致，长时间等待属于任务级决策，主动停止加进度反馈比静默挂起更可预期。

## 报表流程

报表有两族，形态完全不同，分开建模：

**销售与财务报表**是同步下载：单次 GET 返回 gzip 压缩的 TSV，无任务创建、无分页。复杂度在参数组合（厂商编号、报表类型、子类型、频率、日期的合法组合）和权限（个人 key 无法访问；财务报表需要财务类角色）。报表模块负责参数组合校验、解压与解析边界，并对"该日期报表不存在"给出可诊断反馈。

**Analytics 报表**是异步多步流程：创建报表请求（`ONGOING` 持续生成或 `ONE_TIME_SNAPSHOT` 一次性快照）→ 枚举可用报表 → 枚举实例 → 取分段下载地址 → 下载文件。首次请求需 1–2 天才有数据；长期不读取的请求会被 Apple 停止，需要重建。报表模块负责状态推进与轮询细节，业务模块只表达"要哪个报表的哪天数据"。

两族共同的边界：解析结果保留原始字段语义，避免过早做业务映射；权限不足、报表未生成、文件损坏分别给出可诊断错误。

## 媒体上传流程

截图和预览素材上传是 ASC 的标准四步资产流程，由媒体模块整体管理：

1. **预约**——以文件名和字节数创建资产资源，ASC 返回上传计划（一组带 URL、偏移和长度的分片操作）。
2. **传输**——按计划对各分片执行 PUT；分片可并发、可乱序；上传 URL 免认证但限时，失败分片可单独重传。
3. **提交**——标记上传完成并附整文件的 MD5 校验和，字节数必须与预约值一致。
4. **确认**——ASC 异步校验，资产状态机从待上传推进到完成或失败；失败是终态，只能删除资源重来。

媒体模块在失败时保留素材标识、所处阶段和 ASC 侧资产状态（含其中的错误与警告明细），让用户能定位是哪一步、哪个文件的问题。

## 模块协作原则

- 请求层不理解具体报表或媒体业务。
- 工作流模块不直接管理凭据。
- 分页模块不解析业务数据。
- Skill 入口不暴露底层请求细节，只呈现任务级能力和用户可理解的结果（见 [Skill 入口与 CLI 策略](skill-interface.md)）。
