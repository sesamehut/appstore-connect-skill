# 生态调研：ASC SDK 对比

数据采集时间：**2026-06**。社区项目的活跃度和版本状态会变化，引用前需要重新核对。

## 调研结论

ASC SDK 生态的共同点是：多数库围绕 Apple 官方 API 契约生成类型或请求封装，再补一层认证和便利方法。它们的差异主要在语言运行环境、维护节奏、封装厚度和跨平台摩擦。

本项目运行目标是 Windows 和 Linux，Skill 运行时也更贴近 JavaScript 生态。因此项目选择 Node.js/TypeScript，并自生成 API 契约，不把第三方 ASC SDK 作为核心依赖。对应设计判断见 [架构总览](../architecture/overview.md)。

## 生态概况

| 语言方向 | 概况 | 对本项目的判断 |
|---|---|---|
| Swift | 社区成熟度较高，和 Apple 生态贴近 | 不作为默认运行路线，避免增加 Windows/Linux 部署摩擦 |
| TypeScript | 与 Skill 运行环境匹配，有可用社区库 | 采用同语言生态，但核心能力自建 |
| Python | 部分库维护节奏偏慢 | 不作为新实现基础 |

## 方案判断

项目采用的方案是：

- API 契约来自 Apple 官方规范。
- 项目维护自己的认证、请求、分页和多步骤工作流。
- 第三方 ASC SDK 只作为调研参考，不进入核心依赖。

未采用的方案包括直接使用 Swift SDK、直接使用 TypeScript ASC SDK、使用 Python SDK。它们可以在特定环境中成立，但不符合本项目当前的运行目标和维护边界。

## 后续复核点

如果未来出现 Apple 官方 SDK、官方 MCP server，或社区项目形成事实标准，需要重新评估。评估重点应放在：

- 是否降低运行和维护成本。
- 是否改善 API 追新能力。
- 是否保持核心流程可测试、可诊断。
- 是否影响 Windows/Linux 使用体验。
