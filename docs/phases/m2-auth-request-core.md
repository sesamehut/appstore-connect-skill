# M2 认证与请求核心 — 阶段计划

本文是 [Roadmap](roadmap.md) 中 M2 阶段的详细计划。目标：交付所有能力共用的认证与请求地基。策略依据为[认证实现策略](../implementation/authentication.md)与[请求模型与流程策略](../implementation/request-model.md)，Apple 行为核实于 **2026-06**。

## 目标与退出标准

来自 Roadmap：

- 凭据缺失、凭证失效、权限不足、限流四类场景可被区分，并有对应的错误语义。
- 并发触发刷新时复用同一次签发结果。
- 集成测试覆盖认证与请求构造，fixture 不含可用的真实凭据。

本阶段追加一项（经 2026-06-12 与用户确认）：**真实凭据冒烟在 M2 内执行**——签名细节与 Apple 实际行为的偏差只能靠真实环境暴露，越早验证越好。

## 范围与非目标

**范围**：凭据模型与环境变量装载、ES256 凭证签发、凭证复用与保守刷新（含 single-flight 与受控强制重签）、薄请求层（认证注入、查询表达、错误归一、限流透传、退避重试）、覆盖上述行为的集成测试、真实凭据冒烟脚本。

**非目标**（刻意排除，后续阶段按需引入）：

- 不实现分页（M3）、报表与媒体工作流（M5/M6）、Skill 入口（M4）。
- 不为业务资源建立任务级封装；请求层只暴露契约约束的通用调用面，首个业务消费方在 M3 落地。
- 不实现请求级缓存、并发池或主动节流——限流余量透传后，节奏控制属于上层（分页层与工作流层）的职责。
- 错误分类中"文件处理失败"一族不在本阶段（属报表/媒体流程），分类体系为其预留扩展位。

## 决策与理由

技术选型（jose、openapi-fetch、undici MockAgent）的依据见[架构总览](../architecture/overview.md)，本阶段不重新论证；本节只记录 M2 范围内新做出的决策。

**双 key 形态一次到位。** 团队 key 与个人 key 在签发层只差 payload 声明（团队 key 用 Issuer ID，个人 key 用固定的用户主体声明），但凭据校验、错误提示与文档都要覆盖两套形态。经用户确认两者都支持：凭据模型显式区分两种形态，个人 key 在报表、Provisioning 等端点的能力差异不在认证层硬编码，而是留给错误反馈（权限错误提示中说明个人 key 的限制）与 M5 的任务级处理。

**凭据经环境变量装载，key 形态由 Issuer ID 的存在性推断。** Key ID 必填；Issuer ID 存在即团队 key、缺失即个人 key——不引入显式的形态开关变量，因为它在绝大多数配置下是冗余的。误配的代价可控：拼错 Issuer ID 变量名会被当作个人 key 签发并被 Apple 拒绝，认证错误的提示中会说明当前推断出的 key 形态与改正方向。私钥支持文件路径与内联内容两种注入（经用户确认）：路径适合本地开发，内联适合 CI 与容器；两者同时设置按配置错误拒绝，不做静默优先级。

**凭证寿命取 15 分钟，签发时间回拨容忍时钟偏差。** Apple 上限 20 分钟；15 分钟在"复用窗口足够长"与"本机时钟快于 Apple 时仍不触线"之间留出余量。签发时间回拨少量秒数，避免本机时钟略快导致 token 被判定"来自未来"；刷新在到期前的安全余量内提前发生，调用方拿到的凭证保证有最短剩余寿命。

**single-flight 覆盖两条路径：常规刷新与失效重签。** 并发索取凭证时，进行中的签发被全体等待者复用。ASC 判 401 后的受控强制重签以"失效 token"为参数合并：多个并发请求拿同一个失效 token 触发重签时，只有第一个真正重签，其余直接复用结果——避免重签风暴，也避免新签 token 被误判失效再签。401 重签重放只允许一次，再失败按认证错误上抛。

**错误归一采用"抛出类型化错误"，不走返回值分支。** 请求层在响应边界把非成功响应归一为带语义分类的错误并抛出：认证失败、权限不足、资源不存在、参数不合法、限流、外部服务失败、网络失败，每类携带原始 JSON:API errors 数组与（如有）限流快照。选择抛出而非"结果或错误"返回值：归一只发生一次，上层（分页、工作流、Skill 入口）以一致的 catch 语义处理，调用点不必逐个判错分支；成功路径直接拿契约类型的数据。权限错误明确区分于资源不存在（不伪装），并在提示中包含角色与 key 形态线索。

**认证注入走 middleware 接缝，传输重试走自定义 fetch 层，职责分离。** openapi-fetch 的 middleware 是架构总览预留的认证接缝：onRequest 注入凭证，onResponse 处理 401 的单次重签重放与错误归一——认证语义与错误语义集中在一处。429/5xx/网络错误的退避重试属于传输可用性问题，放在传给请求内核的自定义 fetch 包装里：每次尝试克隆请求重发，全抖动指数退避，次数有上限，429 不依赖 Retry-After（ASC 不返回）。分层后 401 重放天然复用传输层的重试与限流解析，互不知晓对方细节。

**限流快照逐响应解析，经观察者回调透传。** 每个响应的 X-Rate-Limit 头被解析为结构化快照；调用方可注册观察者获取每次请求的余量（供 M3 分页与长任务节流），错误对象上也携带当次快照。不把快照塞进成功返回值——那会污染契约类型的数据形态。

**查询序列化按 ASC 规范覆盖默认值。** ASC 的 JSON:API 列表参数（fields、filter 等）按官方规范是逗号拼接的非 explode 形式；openapi-fetch 默认 explode 序列化会产出重复参数名，必须在客户端构造时统一覆盖。这是"查询表达由请求层统一规范"的具体落点。

**M0 占位入口移除。** 工程基础阶段的 URL 拼接占位与其测试被真实的认证与请求模块取代，无向后兼容负担。

**真实凭据冒烟以独立脚本交付，不进入统一检查。** 脚本从环境变量装载凭据，对真实 ASC 发一个最小读请求，输出非敏感的诊断信息（key 形态、响应状态、限流余量）；它需要网络与真实凭据，故不进入 `check` 门，凭据经环境变量注入、不落仓库。

**测试密钥运行时生成，不入库任何静态密钥。** 集成与单元测试所需的 EC P-256 私钥在测试运行时即时生成——即便是无法通过 ASC 认证的假密钥，入库静态 PEM 也会制造 secret-scanner 噪音并培养坏习惯。fixture 由构造保证无法认证：密钥与任何真实 ASC key 无关联。

## 验证清单

- [x] 凭据缺失、私钥双源冲突、私钥文件不可读、PEM 形态不合法分别给出明确错误，错误信息不含凭据值。
- [x] 签发的 JWT 头与 payload 符合 Apple 约束（ES256、kid、aud；团队/个人两种 payload 形态），签名可用对应公钥验证。
- [x] 凭证在安全窗口内复用、临近到期提前重签、寿命不超上限；并发索取只发生一次签发；以失效 token 触发的强制重签合并为一次。
- [x] 401 触发单次重签重放后成功；再次 401 按认证错误上抛。认证失败、权限不足、资源不存在、参数不合法、限流、外部服务失败、网络失败七类错误在集成测试中可区分，原始 JSON:API errors 被保留。
- [x] 429/5xx/网络错误按退避策略重试且次数有上限；4xx（除 429）不重试；限流快照经观察者回调透传并出现在限流错误上。
- [x] 查询参数按 ASC 逗号拼接形式序列化。
- [x] `npm run check` 全绿；测试不含任何静态密钥或真实凭据。
- [ ] 真实凭据冒烟通过：签名链路在真实 ASC 上验证（用户注入凭据，运行一次冒烟脚本）。
