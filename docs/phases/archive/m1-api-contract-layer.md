# M1 API 契约层 — 阶段计划

本文是 [Roadmap](../roadmap.md) 中 M1 阶段的详细计划。目标:把 Apple 官方 OpenAPI 规范变成可控的开发期类型边界,并建立可重复的再生成流程。规范现状核实于 **2026-06**:Apple 以单文件 OpenAPI 3.0.1 JSON 发布(约 6.9 MB,929 个路径、1346 个 schema,版本 4.4),只提供最新版,不保留历史版本。

## 目标与退出标准

来自 Roadmap:

- 契约可以从官方规范一键再生成,结果可复现。
- [架构总览](../../architecture/overview.md) 描述的契约升级三步(更新契约、检查兼容、回归测试)具备可执行基础。

本阶段的首要风险是**生成产物的可读性与评审策略**:官方规范体量大,如果本阶段不定调评审方式,后续每次升级都会退化为人工逐行比对。

## 范围与非目标

**范围**:固定规范获取来源与采集方式、生成流程及其确定性保障、生成产物与元数据的入库与评审策略、产物完整性守卫、规范升版的漂移检测、覆盖生成结果的类型冒烟测试。

**非目标**(刻意排除,后续阶段按需引入):

- 不实现任何运行时行为——认证、请求层、错误归一属于 M2;本阶段产物是纯类型,运行时体积为零。
- 不为契约类型建立手写的再导出或包装层;消费方式(请求内核直接消费 `paths` 类型)在 M2 随真实调用点落地。
- 不按产品首期范围裁剪规范(决策与理由见下节)。

## 决策与理由

工具选型(openapi-typescript v7、零运行时单文件产物)的依据见 [架构总览](../../architecture/overview.md) 与 [代码生成与运行时栈调研](../../research/codegen-and-runtime-stack.md),本阶段不重新论证;本节只记录 M1 范围内新做出的决策。

**全量生成,不按首期范围过滤。** 生成器没有内置路径过滤,裁剪意味着自建规范切割工具,并持续维护资源清单与 schema 引用闭包——每次里程碑扩范围都要先改过滤器。产物是纯类型,运行时成本为零,全量的代价只是开发期检查耗时与仓库体积,一次性可控;换来的是 M5–M9+ 扩能力时契约零改动。

**规范快照不入库,只入库元数据。** 与产物同目录提交一份清单,记录规范版本、来源地址、采集时间、规范哈希与产物哈希;下载的规范本体只存在于本地缓存目录,不进仓库。生成产物本身已是入库的记录性工件,规范本体再入库则是约 7 MB 的冗余。代价是历史复现依赖产物的 git 历史而非重新获取(Apple 只服务最新版);社区规范存档仓库(见 [调研](../../research/codegen-and-runtime-stack.md))可作为追溯时的旁证,不作为依赖。

**确定性生成与防手编守卫。** 产物是规范字节、生成器版本与生成选项的纯函数:生成头注释只含内容派生信息(生成器版本、规范版本与哈希),不含时间戳;输出按字母序排序,Apple 重排规范条目时升级 diff 只反映语义变化;规范哈希未变时采集时间保留原值——同一输入重复生成必然零 diff。统一检查入口新增一道最便宜的前置门:校验入库产物哈希与清单一致、清单字段完整、记录的生成器版本与当前安装版本一致。任何对产物的手工修改、清单与产物的脱节、生成器升级后未再生成,都会在本地与 CI 的同一道门上失败。

**生成产物的工具豁免。** 生成目录约定为 100% 生成内容,手写文件(含测试)不得进入;lint、格式化、覆盖率统计按目录整体排除——产物的字节稳定性不能受格式化工具版本影响,类型感知 lint 在十万行级纯类型文件上也只有成本没有收益。仓库属性将该目录标记为生成代码,代码平台审查时默认折叠、按需展开。

**评审策略:冒烟断言 + 折叠 diff。** 评审不逐行读产物,而是依赖三层:类型冒烟测试锚定关键资源(应用列表操作的响应形态、JSON:API 查询参数面、资源类型字面量、错误响应形态),空生成或生成模式漂移直接变成类型错误;升级时按字母序排序后的折叠 diff 反映语义变化面;清单中的版本与哈希提供变化的事实记录。

**漂移检测纳入本阶段。** Apple 不保留历史版本,发现升版越晚,丢失的 diff 粒度越不可追回。CI 增加一个独立的定时作业,每周获取最新规范并与清单中的哈希比对,不一致即失败并提示升级流程。该作业与统一检查门互不影响:检查门完全离线,漂移作业才接触网络。

**生成脚本形态。** 获取、生成、校验、漂移检查四个命名入口,职责单一、可独立运行;获取与生成分离,使"换生成选项重新生成"无需重新下载,也使漂移检查可以复用获取入口。脚本以工程脚本形式存在,不进入运行时源码,也不参与构建产物。解压 Apple 的 zip 分发需要一个零依赖的小型开发期依赖(Node 无内置 zip 支持,跨平台 shell 方案在 CI 矩阵上不成立)。

## 验证清单

- [x] 全新克隆 + 依赖安装后,一条命令完成"获取规范 → 生成契约 → 写入清单",统一检查通过。
- [x] 同一规范输入重复生成,工作区零 diff(确定性成立)。
- [x] 破坏性验证:产物单字节篡改只让完整性门失败;从冒烟测试覆盖的资源中移除路径或 schema 只让类型检查失败。
- [x] 清单记录规范版本、来源地址、采集时间与双哈希(Roadmap 设计要点)。
- [x] lockfile 全部依赖解析自公共 registry;CI 矩阵全绿。
- [x] 漂移检测作业手动触发一次通过。

实施期间的三个值得记录的事实:生成器的 peer 依赖仍声明 TypeScript 5,与仓库的 TypeScript 6 经包管理器 overrides 声明式对齐,全部检查实证兼容;十三万行纯类型产物对统一检查的耗时影响轻微(全程约 7 秒),计划中的备选方案未启用;Windows 本机生成的产物在 Linux 与 macOS CI 上哈希一致,跨平台字节稳定性成立。
