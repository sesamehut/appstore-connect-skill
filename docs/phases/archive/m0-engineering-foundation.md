# M0 工程基础 — 阶段计划

本文是 [Roadmap](roadmap.md) 中 M0 阶段的详细计划。目标:让仓库具备开发、测试和验证代码的最小工程能力。工具链版本核实于 **2026-06-12**(npm registry / GitHub Releases)。

## 目标与退出标准

来自 Roadmap:

- 仓库根目录可以一键运行全部检查(类型检查、lint、格式检查、测试),且 CI 在主分支与 PR 上执行同样的检查。
- 测试框架支持集成风格测试,外部边界可以在清晰的接缝处 mock。

本阶段的首要风险是**过度配置**。所有取舍都向"够用"倾斜:不为尚不存在的代码预设结构,不引入当前没有消费方的机制。

## 范围与非目标

**范围**:TypeScript/ESM 工程骨架、统一检查入口、测试框架接入、HTTP 边界 mock 接缝验证、跨平台一致性策略、CI 基本流水线、一个最小纵向切片(真实源码 + 单元测试 + 集成风格测试)用来证明整条链路。

**非目标**(刻意排除,后续阶段按需引入):

- 零运行时依赖——jose 与 openapi-fetch 属于 M2,契约生成属于 M1。
- 不预建 [架构总览](../architecture/overview.md) 描述的六层目录结构;目录随真实代码出现。
- 不引入 git hooks(提交前检查)。检查统一收口在一个入口命令与 CI,提交动作不被阻塞;后续如有需要随时可加。
- 不引入发布、版本管理、依赖机器人等工具链。

## 工具链决策与理由

运行时、语言、测试框架的选型依据见 [架构总览](../architecture/overview.md) 与 [代码生成与运行时栈调研](../research/codegen-and-runtime-stack.md),M0 不重新论证;本节只记录 M0 范围内新做出的决策。

**包管理:npm + 提交 lockfile。** [Skill 入口与 CLI 策略](../implementation/skill-interface.md) 的安装路线已经锚定 npm 与确定性依赖树,仓库开发期沿用同一工具,避免双工具链。

**类型检查与产出分离(双 tsconfig)。** 一份配置面向检查、编辑器与类型感知 lint,覆盖源码、测试与工程配置文件,不产出任何文件;另一份只负责把 src 编译为 ESM 加类型声明,排除共置测试。理由:检查范围应当大于产出范围(测试与配置文件也要被类型检查),把两个关注点揉进一份配置会互相牵制。严格选项(索引访问未检查告警、可选属性精确语义、显式 override 等)在仓库零代码时全部开启——此刻采纳零成本,事后补成本极高。

**Lint:ESLint flat config + typescript-eslint 类型感知严格规则 + Prettier 负责格式。** 本项目是异步密集型代码(HTTP、JWT、分页、多步工作流),类型感知规则(如悬空 Promise 检测)能抓到真实缺陷,这是 Biome / oxlint 等更快的替代品在 2026-06 仍覆盖不全的能力,故以规则完整性优先于执行速度。格式问题全部交给 Prettier,以禁用冲突规则的预设殿后,lint 只管正确性。Prettier 不接管 Markdown:中文文档的 CJK 排版与手工格式由作者负责。生成契约产物的 lint 排除推迟到 M1 路径真实存在时再加。

**测试:Vitest + undici MockAgent。** 集成风格测试直接拦截全局 fetch,并禁用真实网络连接,使"外部边界在清晰接缝处 mock"成为被测试证明的事实而不是口头约定。约束:undici 作为开发依赖,其主版本须与开发/CI 主力 Node 内置的 undici 主版本对齐;运行基线 Node 22 内置更老的主版本,依赖全局 dispatcher 符号的跨版本稳定性,由 CI 的基线作业实证把关。

**统一检查入口。** 单条命令顺序执行类型检查 → lint → 格式检查 → 测试,失败成本低的在前。本地与 CI 跑同一条命令,不存在"本地绿 CI 红"的口径分叉。不引入并行任务运行器:四个串行步骤的输出可读性优于并行交错,且少一个依赖。

## 跨平台一致性策略

开发机在 Windows,CI 覆盖三个 OS,行尾是最大的一致性风险源:

- 仓库层面强制所有文本文件以 LF 入库与检出(.gitattributes),压过本机 autocrlf 配置;Prettier 的默认行尾正是 LF,两者闭环后格式检查在任何平台结果一致。
- 编辑器层面用 .editorconfig 声明编码、行尾、缩进与行尾空白规则,作为与具体编辑器无关的兜底。
- 不引入 .nvmrc / .node-version:Node 版本期望已由 engines 声明、CI 矩阵与 README 三处表达,再加一处是漂移面而非保险。

## CI 设计

GitHub Actions,push 到主分支与全部 PR 触发同一作业,执行与本地完全相同的统一检查命令;同一分支的新推送取消进行中的旧运行。

矩阵共 4 个作业,按"两条独立风险轴各自铺开"设计,而不是全组合:

| 作业 | 验证什么 |
|---|---|
| ubuntu × Node 22 | engines 运行基线;mock 工具对基线 Node 内置 fetch 的跨主版本兼容 |
| ubuntu × Node 24 | 开发与 CI 主力线 |
| windows × Node 24 | 路径、行尾策略、脚本串联(开发机平台) |
| macos × Node 24 | ASC 用户主力平台 |

Node 版本风险与 OS 无关,在最便宜的 ubuntu 上铺开两档;OS 风险与 Node 版本无关,各取主力一档。全组合多出的作业不带来新的风险覆盖。作业之间不级联失败,便于一眼定位平台特有问题。

回退预案:若基线作业上 mock 拦截不成立(内置 undici 跨主版本差异),该作业降级为类型检查 + lint + 构建并在本文记录约束,不为开发期问题抬高 engines 基线。

## 最小纵向切片

为了让每个检查门都有真实对象,M0 落一个最小但非一次性的切片:ASC API 基础地址常量与一个 URL 拼接函数,配一条共置单元测试和一条集成风格测试。集成测试即上文的 mock 接缝证明。M2 请求层落地时允许重塑这部分代码;它的价值在于证明链路,不在于形态稳定。

## 验证清单

- [ ] 全新依赖安装后,统一检查命令一次通过。
- [ ] 逐门破坏性验证:类型错误、悬空 Promise、格式破坏、断言翻转,各自让对应的门失败且只让该门失败。
- [ ] 构建产物只含源码编译结果(不含测试),可被 Node 直接 import。
- [ ] 集成测试在禁用真实网络连接的前提下通过。
- [ ] Windows 工作区格式化后无残留 diff(LF 策略成立)。
- [ ] PR 上 4 个矩阵作业全绿;合入后主分支运行全绿。
