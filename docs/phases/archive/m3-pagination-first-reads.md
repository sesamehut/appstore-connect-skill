# M3 分页与首批读能力 — 阶段计划

本文是 [Roadmap](../roadmap.md) 中 M3 阶段的详细计划。目标:用真实读取场景验证 M0–M2 地基,交付第一条端到端链路。策略依据为[请求模型与流程策略](../../implementation/request-model.md)的分页策略一节与[架构总览](../../architecture/overview.md)的分层思路,Apple 行为核实于 **2026-06**。

## 目标与退出标准

来自 Roadmap:

- 业务调用不接触游标即可读取多页数据。
- 应用与版本读取能力在真实账号下冒烟通过。
- 权限不足、资源不存在等错误路径的反馈符合错误分类策略。

## 范围与非目标

**范围**:分页模块(游标续读、单页/限量/全量三态读取语义、失败页进度上下文保留、限流余量地板保护);首批读能力切片——应用列表、应用详情、应用版本列表;覆盖上述行为的集成测试;真实凭据冒烟脚本扩展。

**非目标**(刻意排除,后续阶段按需引入):

- 不做 bundleId 便捷查找、版本详情读取与 include 关系展开——能力面在 M4 随元数据与评论能力一起扩展(经 2026-06-12 与用户确认,本阶段取最小验证切片)。
- 不做自适应页间延迟、并发预取或请求级缓存——长时间等待属于任务级决策,留给工作流层(M5)与 Skill 入口(M4)。
- 不做写能力与任务级反馈口径,Skill 入口形态在 M4 落地。

## 决策与理由

**分页核心是页级异步生成器,游标处理百分之百内部化。** 分页是控制流而非单个请求的属性:核心以惰性生成器逐页产出契约类型的页文档,消费方提前终止(限量读取、break)天然不再多发请求。在其上提供收集助手实现三态读取语义——单页、限量条目、全量;能力层与未来的工作流层都以"声明语义"而非"操作游标"的方式消费。备选的"调用方提供取页闭包"形态被否决:它把每个端点的类型组装与游标续传细节推回调用方,恰好泄漏给了最不该接触游标的那一层。

**`links.next` 只取查询串、原样透传。** 续页请求复用原 path 与路径参数,查询串从 next 链接整体取出、逐字节原样重发,不解析也不重组游标。这样既维持游标的不透明契约,又让 client 上配置的 baseUrl 覆盖(测试与代理场景)在续页上继续生效。空查询串与重复出现的相同查询串按上游异常上抛——前者意味着无游标的 next 链接会造成死循环,后者是零成本的死循环保险丝。

**读取语义三态、scope 必填。** 列表能力的调用方必须显式声明单页、限量还是全量——读取成本永远是调用点的显式决策,不藏在默认值里。这是"业务模块只声明需要单页、限量还是全量数据"的直接落点。

**全量读取以限流余量地板保护节奏(经 2026-06-12 与用户确认)。** 分页层逐页解析 `X-Rate-Limit` 余量;仍有下一页且余量低于地板值时主动停止,按限流语义抛出专用错误,携带已读页数与条目数。地板检查发生在当前页产出之后——已花配额换来的数据不丢弃。默认地板值在小时配额(默认 3500)中留出交互与重试余量,可按调用配置,置零即关闭。不做静默自适应延迟:429 不带 `Retry-After`,等满窗口可能接近一小时,长等待是任务级决策,主动停止加进度反馈比静默挂起更可预期。

**失败页上下文以附加进度切面实现,不包裹新错误类。** 第 N 页失败时,错误本身仍是请求层归一出的那一类(权限不足就是权限错误),分页层只把"已读页数/条目数"作为切面附加到原错误实例后重抛。错误分类、`instanceof` 判断与原始 JSON:API errors 的既有契约零侵蚀;包裹式新错误类会迫使所有上层多学一种类型并破坏子类判断。地板保护错误是限流错误的子类,分类语义不变,同时可与真实 429 区分。

**能力层作为第七个概念层落位,形态为函数式薄面。** 首个业务消费方在本阶段落地,架构分层表在分页层与工作流层之间新增能力层:把契约资源组织成可直接调用的业务读写操作,与多步骤状态推进的工作流层明确区分。能力以独立函数交付、client 作首参:无状态可封装(client 已组合认证/重试/限流)、可摇树(M8 单文件打包)、与既有函数式风格一致;M4 的 CLI 持有一个 client 分发到这些函数即可。

**能力选项类型自契约 operations 派生,返回原样条目加分页诊断信封。** 过滤、字段、排序选项的类型直接派生自生成契约的 operation 查询参数——契约升级自动流动到能力面,不可能漂移。列表返回"契约类型条目数组 + 已读页数/总量估计/是否截断/最新限流快照"的信封:条目不做业务映射,诊断信息供 M4 给出诚实的任务反馈(如"读取了 200 条,共约 1500 条,已按限量截断")。应用详情返回完整响应文档而非裸资源,M4 增加 include 支持时只扩展不重塑。

**版本状态过滤采用 `appVersionState`。** 契约已将 `filter[appStoreState]` 标记为 deprecated 并指向 `filter[appVersionState]`;新建的能力面不应背着首日即弃用的参数。

**可分页端点由响应结构过滤,不依赖业务 schema。** 类型层以"页文档形态"(数据数组加 next 链接)在契约的全部 GET 端点上做结构过滤,得到可分页路径的联合类型;集合端点天然命中、详情端点天然被拒,分页模块对具体业务资源零知识。

## 验证清单

- [x] 多页全量读取按序产出全部条目,请求数与页数一致;续页请求的查询串与 next 链接逐字节一致。
- [x] 单页语义恰好一个请求;限量语义在页中与页界提前停止且不多发请求;首请求无查询参数时续页正常。
- [x] 页中失败(如第 3 页 403)上抛权限错误,分类与原始 errors 保留,并携带已读页数/条目数;消费方保有已读页面。
- [x] 低余量触发地板保护:抛限流语义的专用错误(含地板值、进度、最新快照),不再发起下一页请求;置零关闭;末页低余量正常完成;响应无限流头不触发。
- [x] 畸形 next 链接、空查询串、重复游标按上游异常上抛。
- [x] listApps 过滤/字段/排序按 ASC 逗号拼接形式序列化;跨页全量读取经能力层端到端可用。
- [x] getApp 路径替换与字段选择正确;404 归一为资源不存在。
- [x] listAppStoreVersions 平台与版本状态过滤可用;403 归一为权限不足,与 404 可区分。
- [x] 类型层断言:集合端点可分页、详情端点被类型系统拒绝;typecheck 耗时无明显劣化(实测 2.7s → 4.7s,结构过滤一次性成本,在开发环路可接受量级)。
