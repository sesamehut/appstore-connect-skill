# 项目 Roadmap

本文是 appstore-connect-skill 从设计阶段走向可分发 Skill 的阶段规划。能力边界以 [产品范围](../product/api-scope.md) 为准，分层与模块职责以 [架构总览](../architecture/overview.md) 为准。本文只回答三件事：先做什么、后做什么、每一步做到什么程度算完成。

规划基准时间：**2026-06**。阶段推进时如发现 Apple API 行为或生态变化，先复核研究笔记和官方文档，再调整本文。

## 推进原则

- **纵向切片优先**——每个阶段以"一条端到端可验证的链路"收尾，而不是按层批量铺设。请求层、分页、工作流的设计判断都需要真实能力来检验。
- **地基先行，能力跟进**——契约、认证、请求核心是所有能力的公共地基，必须先稳定；业务能力按产品首期范围分批落地。
- **Skill 形态早验证**——在首批读能力可用时就引入最小 Skill 入口，避免最后才发现任务组织或反馈形式不成立。
- **阶段文档生命周期**——每个阶段开始时在本目录建立该阶段的详细计划，完成后移入 `archive/`；本文只保留里程碑级别的状态。

## 里程碑总览

| 阶段 | 主题 | 核心产出 | 状态 |
|---|---|---|---|

| M1 | API 契约层 | 从 Apple 官方规范生成的契约与再生成流程 | 进行中（[阶段计划](m1-api-contract-layer.md)） |

| M4 | 元数据与评论能力、最小 Skill 入口 | 资源读写能力面与 Skill 雏形 | 未开始 |
| M5 | 报表工作流 | 销售/财务/分析报表下载链路 | 未开始 |

依赖关系大体线性：M0 → M1 → M2 → M3 是地基链，必须按序完成；M4 之后的业务能力都依赖 M3，彼此可按价值调序，其中 M5 与 M6 互相独立，可并行或交换顺序。

## M0 工程基础

**目标**：让仓库具备开发、测试和验证代码的最小工程能力。

**范围**：TypeScript/ESM 工程骨架、测试框架选型与接入、统一的项目检查入口（类型检查、lint、测试）、CI 基本流水线。

**退出标准**：

- 仓库根目录可以一键运行全部检查，且 CI 在主分支与 PR 上执行同样的检查。
- 测试框架支持集成风格测试，外部边界可以在清晰的接缝处 mock。

本阶段的风险是过度配置：工具链以够用为准，不为尚不存在的代码预设结构。

## M1 API 契约层

**目标**：把 Apple 官方 OpenAPI 规范变成可控的开发期类型边界。

**范围**：固定官方规范的获取来源、选择生成方式、建立可重复的再生成流程、约定生成产物的存放与评审方式。

**设计要点**：生成产物是边界工件，只约束资源、参数和响应形态，不承载业务逻辑；同一规范输入必须产出一致结果，规范版本与采集时间需要被记录。

**退出标准**：

- 契约可以从官方规范一键再生成，结果可复现。
- [架构总览](../architecture/overview.md) 描述的契约升级三步（更新契约、检查兼容、回归测试）具备可执行基础。

**风险**：官方规范体量大，生成产物的可读性与评审策略需要在本阶段定调，避免后续每次升级都依赖人工逐行比对。

## M2 认证与请求核心

**目标**：交付所有能力共用的认证与请求地基。

**范围**：按 [认证实现策略](../implementation/authentication.md) 实现凭证签发、复用与保守刷新；按 [请求模型](../implementation/request-model.md) 实现薄请求层——认证注入、查询表达、错误归一、限流信息透传。

**退出标准**：

- 凭据缺失、凭证失效、权限不足、限流四类场景可被区分，并有对应的错误语义。
- 并发触发刷新时复用同一次签发结果。
- 集成测试覆盖认证与请求构造，fixture 不含可用的真实凭据。

**风险**：签名细节与 Apple 实际行为的偏差只能靠真实环境暴露，需要安排一次真实凭据冒烟验证；凭据经环境变量注入，不进入仓库。

## M3 分页与首批读能力

**目标**：用真实读取场景验证地基，交付第一条端到端链路。

**范围**：分页模块（游标续读、单页/限量/全量语义、失败页上下文保留）；以应用列表、应用详情和版本信息读取作为验证切片。

**退出标准**：

- 业务调用不接触游标即可读取多页数据。
- 应用与版本读取能力在真实账号下冒烟通过。
- 权限不足、资源不存在等错误路径的反馈符合错误分类策略。

## M4 元数据与评论能力、最小 Skill 入口

**目标**：把读能力扩展为读写能力面，并第一次以 Skill 形态暴露。

**范围**：版本元数据与本地化的读取和更新；评分与评论读取、开发者回复；覆盖既有能力的最小 Skill 入口与任务组织雏形。

**设计要点**：Skill 入口此时只需证明 agent 能以任务语义调用能力并得到可读反馈，能力清单不求全；输出必须区分"Apple API 不支持"与"本项目暂未实现"，这是 [产品范围](../product/api-scope.md) 的硬性设计要求。

**退出标准**：

- 元数据与本地化更新链路在测试账号验证通过，评论读取与回复可用。
- 通过 agent 实际调用 Skill 完成一次真实任务，例如读取应用元数据并修改一处文案。

## M5 报表工作流

**目标**：交付第一个多步骤工作流，验证工作流层设计。

**范围**：销售/财务报表与 Analytics 报表的任务创建、状态推进、文件下载与解析边界；权限不足、报表未生成、文件损坏等可诊断错误。

**退出标准**：

- 报表从发起请求到本地结构化结果全链路可用。
- 轮询与状态推进细节不泄漏给业务调用方。
- 个人 key 与团队 key 的权限差异有明确反馈。

## M6 媒体素材工作流

**目标**：交付截图与预览素材上传链路。

**范围**：上传上下文创建、按 ASC 返回的上传计划推进传输、完整性校验与提交确认；失败时保留素材、阶段与 ASC 资源状态。

**退出标准**：

- 单张截图与整组素材上传在测试应用验证通过。
- 流程中断后可以诊断出停在哪个阶段。

## M7 TestFlight 与送审准备

**目标**：补齐产品首期范围的最后两块能力，形成能力闭环。

**范围**：TestFlight 基础管理（测试组、测试员、构建分发可见性、反馈读取）；送审准备与发布配置（送审材料检查、版本提交相关资源）。build 二进制上传明确不在范围内，TestFlight 能力围绕已存在的构建展开。

**退出标准**：

- [产品范围](../product/api-scope.md) 首期列出的六项能力全部可经 Skill 调用。
- 每项能力有任务级反馈与错误提示。

## M8 Skill 完备化与发布

**目标**：把"能力可用"打磨成"可分发的 Skill"。

**范围**：任务组织与描述打磨、输入校验、用户可读反馈的统一口径、使用文档（配置与凭据说明）、分发打包（单文件 CLI 产物与 plugin/marketplace 形式，见 [Skill 入口与 CLI 策略](../implementation/skill-interface.md)的运行环境保障）。

**退出标准**：

- 新用户按文档可以从零完成配置并跑通一个真实任务。
- Skill 描述准确反映能力边界，agent 不会被误导去调用不存在的能力。
- 分发包安装后无需执行任何依赖安装步骤即可运行，环境前提仅为 Node。

## M9+ 扩展与持续事项

不设固定排期，按需求拉动：

- **扩展能力**——商业化资源（内购、订阅、价格、优惠）、营销资源（自定义产品页、应用内事件）。
- **受限能力**——组织与访问、Provisioning。
- **后续集成**——Xcode Cloud、Webhooks、EU 替代分发。
- **持续事项**——Apple 规范升级按 [架构总览](../architecture/overview.md) 的升级策略执行；当官方 SDK、官方 MCP server 或社区事实标准出现时，按 [SDK 对比](../research/sdk-comparison.md) 与 [生态调研](../research/mcp-skill-landscape.md) 列出的复核点重新评估。
