# 架构总览

本文描述目标架构与技术选型。工程基础与 API 契约层已落地（见 [Roadmap](../phases/roadmap.md)），运行时行为代码自认证与请求核心阶段起逐步交付。文中的 Apple 行为与生态状态核实于 **2026-06**；选型的对比证据保存在 [代码生成与运行时栈调研](../research/codegen-and-runtime-stack.md)。

## 背景与约束

本项目要把 App Store Connect 的高频操作整理成 Claude Skill 可调用的能力。它面对的不是单一 macOS 开发环境，而是包含 Windows 和 Linux 的日常自动化环境，因此架构首先要降低运行门槛，避免把平台工具链变成使用前置条件。

ASC 的资源、请求参数和响应形态由 Apple 官方 OpenAPI 规范定义（2026-06 当前版本 4.4，约 930 个 path、1300+ schema），并随官方能力持续变化。项目把官方规范作为资源模型的唯一来源；同时把认证、分页、错误处理和多步骤文件流程留在自己手里，不交给第三方 ASC SDK。

Skill 的入口面向用户任务，而不是底层接口调用。架构需要把"读取应用信息""推进版本元数据""下载报表""上传素材"这类意图组织成稳定能力，并把外部 API 的权限、限流和错误反馈转化为可诊断的信息。

## 架构目标

- 在 Windows 和 Linux 环境中低摩擦运行。
- API 契约跟随 Apple 官方规范，减少手写接口定义。
- 运行时保持轻量、可审计，依赖面刻意压到最小。
- 普通资源请求和多步骤文件流程分开，降低复杂度扩散。
- 对凭据、权限、限流和错误保持明确边界，不隐藏外部 API 的不确定性。

## 技术选型

| 关注点 | 选型 | 一句话理由 |
|---|---|---|
| 运行时 | Node.js（基线 ≥22.12，主要目标 24 LTS）+ TypeScript 6.x，ESM-only | 跨平台、零额外工具链 |
| API 契约 | openapi-typescript v7（types-only 生成） | 确定性单文件类型产物，零运行时 |
| HTTP 内核 | Node 原生 fetch + openapi-fetch | 直接消费契约类型，调用点自动推断 |
| JWT 签名 | jose v6 | ES256 与 Apple .p8 私钥开箱即用 |
| 测试 | Vitest 4 + undici MockAgent | TS/ESM 零配置，以原生 fetch 同方言 mock 边界 |
| 工程检查 | ESLint（flat config）+ typescript-eslint 类型感知严格规则 + Prettier | 异步密集代码靠类型感知规则兜底，格式全权交给 Prettier |
| 构建 | 开发态纯 tsc 产出 ESM；分发态 tsdown 打单文件 CLI | 开发最可审计，分发零安装步骤 |

### 运行时与语言

采用 Node.js/TypeScript、ESM-only。Swift 天然贴近 Apple 生态，但会把运行环境推向 macOS 和额外工具链；本项目更看重跨平台执行、脚本式交付和 Skill 集成的低摩擦体验。Node 基线定在 22.12（该点位起 ESM 包可被 CJS 直接 require），开发与 CI 以 Node 24（Active LTS）为主要目标；Node 20 已于 2026-04 EOL，不支持。TypeScript 钉在 6.x 稳定线，待基于 Go 的 TypeScript 7 发布稳定版后再评估切换。

### API 契约：openapi-typescript

契约层从 Apple 官方 OpenAPI 规范生成，工具选 openapi-typescript（7.x 稳定版）。它只生成类型、不生成运行时代码，输出是单个确定性的类型声明文件——这与"契约只做类型边界、运行时全手写"的架构定位完全同构：生成产物可提交入库，规范升版时靠 diff 审查变化；默认把 enum 内联为字符串字面量联合，避开了 ASC 规范中重名 enum 导致部分生成器失败的已知问题。

备选方案 @hey-api/openapi-ts 功能更全（可生成 SDK 函数与运行时校验），但仍处 0.x、破坏性变更频繁，多文件产物会放大契约 diff 噪音，不符合"契约产物作为稳定可审查边界"的诉求。

契约对整份官方规范**全量生成**，不按产品首期范围裁剪：生成器没有内置路径过滤，裁剪意味着自建规范切割工具并持续维护资源清单与 schema 引用闭包，每次扩能力范围都要先改过滤器；产物是纯类型，全量的代价只是开发期检查耗时与仓库体积（实测均在可忽略量级），换来的是后续里程碑扩能力时契约零改动。

生成被约束为规范字节、生成器版本与生成选项的**纯函数**：头注释只含内容派生信息（不含时间戳），输出按字母序排序，同一输入重复生成必然零 diff，规范升版的 diff 也只反映语义变化而非条目重排。产物旁随附一份元数据清单，记录规范版本、来源地址、采集时间与规范/产物双哈希；统一检查入口的第一道门离线校验产物与清单的一致性，以及清单记录的生成器版本与当前安装版本的一致性——手工修改产物、或生成器升级后未再生成，都会在本地与 CI 的同一道门上失败。生成目录整体豁免 lint、格式化与覆盖率统计，并在仓库属性中标记为生成代码以折叠评审 diff；对关键资源形态的类型冒烟断言代替逐行评审，空生成或生成模式漂移直接表现为类型检查失败。

### HTTP 内核：原生 fetch + openapi-fetch

HTTP 调用基于 Node 原生 fetch，不引入 axios/got 这类完整客户端——薄请求层需要的能力它们大多用不上，而依赖面本身就是风险（2026-03 的 axios 供应链投毒事件进一步强化了"少依赖"的社区共识）。

在原生 fetch 之上采用 openapi-fetch 作为请求内核：它是 openapi-typescript 的官方配套（约 6KB、零业务逻辑），直接消费生成的契约类型，让每个调用点自动获得参数与响应的类型推断，并通过 middleware 机制为认证注入留出清晰接缝。它是通用 fetch 封装而非 ASC SDK，不违反"不依赖第三方 ASC SDK"的边界。备选是完全手写这一层，但那意味着自己维护一套基于契约类型的泛型推断管道——复杂度集中在类型体操而非业务行为，不值得自建。

### 认证签名：jose

ASC 要求 ES256 签名的 JWT。签名库选 jose（6.x）：零依赖、ESM-first、与 WebCrypto 对齐，且能直接导入 Apple 分发的 .p8（PKCS#8）私钥文件，无需格式转换。曾经的事实标准 jsonwebtoken 已近乎停止维护且仅支持 CJS，不进入考虑。认证策略详见 [认证实现策略](../implementation/authentication.md)。

### 测试与构建

测试框架选 Vitest 4：对 TypeScript ESM 项目零配置，coverage、watch、mock 均为稳定能力，匹配本项目"集成优先"的测试策略；Node 内置 test runner 的 coverage 与 module mock 在 2026-06 仍是实验状态，不适合测试密度高的项目。HTTP 边界统一用 undici MockAgent mock——它与被测代码说同一种方言（原生 fetch 的底层实现），不在请求层和"网络"之间引入额外抽象。

工程检查采用 ESLint（flat config）加 typescript-eslint 的类型感知严格规则，格式问题全权交给 Prettier。这组取舍以规则完整性优先于执行速度：本项目是异步密集型代码，悬空 Promise 这类缺陷只有类型感知规则能稳定抓住，而更快的单工具替代品（Biome、oxlint）在 2026-06 对此类规则的覆盖仍不完整。类型检查、lint、格式检查与测试收敛为一个统一检查入口，本地与 CI 执行同一条命令，不存在口径分叉。

构建分两个形态。**开发态**用纯 tsc 产出 ESM 加类型声明，不引入打包器：逐文件产物与源码一一对应，最可审计，日常迭代与测试都基于它。**分发态**（对外发布的 Skill 包）则用 tsdown 把 CLI 连同依赖打成单个自包含 JS 文件：Claude Code 安装 Skill 只是放置目录，不存在依赖安装钩子，若分发产物要求用户机器执行 npm install，等于把安装失败留到运行期；单文件产物把环境前提收缩到"有 Node"一项。tsdown 是已停止维护的 tsup 的官方继任者，仍处 0.x 须钉版本——它只出现在发布打包一步，不进入日常开发环路，风险面可控。

### 依赖纪律

运行时依赖保持两个：jose 和 openapi-fetch。任何新增依赖如果改变结构、运行时行为或信任边界，先在本文补充设计判断，再进入代码。

## 分层思路

系统分为六个概念层：

| 层 | 职责 | 实现方式 |
|---|---|---|
| API 契约层 | 约束资源、参数和响应形态 | 从官方规范生成，不手编 |
| 认证层 | 管理 ASC 凭据、签发短期凭证、控制刷新 | 手写，基于 jose |
| 请求层 | 基础请求、认证注入、查询表达、错误归一、限流透传 | 手写，内核为 openapi-fetch |
| 分页层 | 把游标分页抽象为连续读取流程 | 手写 |
| 工作流层 | 报表下载、媒体上传等多步骤流程 | 手写 |
| Skill 入口层 | 把能力组织成 agent 可调用的任务 | 手写，形态为单一 CLI |

## Skill 入口形态

Skill 入口采用**单一 CLI + 子命令**：一个可执行入口，按用户任务组织子命令（应用与版本、元数据、TestFlight、报表、媒体、评论等），agent 以黑盒方式调用并通过自描述帮助了解参数。选择单一 CLI 而非一组离散脚本，是为了把参数校验、错误口径和输出格式收敛在一处，能力增长时入口不线性膨胀；Anthropic 官方 claude-api skill 的子命令注册表是这一形态的直接先例。SKILL.md 承担路由职责，具体设计见 [Skill 入口与 CLI 策略](../implementation/skill-interface.md)。

## 核心数据流

普通资源请求：

1. 调用方提出业务意图，例如读取应用信息或更新版本元数据。
2. 请求层依契约类型组织请求，并向认证层获取有效凭证。
3. ASC 返回标准资源响应；错误被归一为带语义分类的诊断信息。
4. 如果存在分页，分页层继续读取后续页。
5. 调用方获得统一整理后的结果或可诊断错误。

多步骤文件流程：

1. 工作流层先创建或定位 ASC 侧的任务资源。
2. 按 ASC 返回的状态推进轮询、上传或下载。
3. 文件处理逻辑独立于普通资源请求，避免污染通用请求层。
4. 最终输出结构化结果，并保留足够的诊断信息。

报表与媒体上传的具体流程模型见 [请求模型与流程策略](../implementation/request-model.md)。

## 设计取舍

项目选择"官方 API 契约 + 薄运行时封装"的路线：把变化最频繁的 ASC 资源定义交给官方规范，把项目真正需要控制的行为集中在认证、分页、错误和多步骤流程上。这个取舍的核心不是减少代码量，而是保持边界清楚——契约层跟随官方，运行时层手写维护，Skill 层只表达用户任务。

在"零依赖纯手写"与"借助成熟生态组件"之间，项目选择引入极少量通用组件（jose、openapi-fetch），但坚持两条线：组件必须是通用基础设施而非 ASC 专用封装；信任边界相关的行为（凭证生命周期、错误分类、重试决策、流程推进）必须留在手写代码里。

## 规格升级策略

Apple 仅发布最新版规范（单文件 OpenAPI 3.0.1 JSON，无历史归档）。所用规范的版本号、采集时间与哈希记录在契约产物旁的元数据清单中，生成产物提交入库；规范本体不入库——产物本身已是入库的记录性工件，历史追溯依赖它的 git 历史，社区规范存档仓库仅作旁证而非依赖。CI 以每周定时任务获取最新规范并与清单哈希比对，升版即时报警：Apple 不留历史，发现越晚，丢失的 diff 粒度越不可追回。升级流程：

1. 从官方固定来源获取新规范，重新生成契约，靠 diff 审查变化面。
2. 检查既有请求和工作流是否仍符合新契约。
3. 运行测试，重点关注认证、查询表达、分页和文件流程。

如果升级导致模块职责、依赖方向或运行策略变化，需要在本文补充对应设计判断，避免只改实现而不更新约束。
